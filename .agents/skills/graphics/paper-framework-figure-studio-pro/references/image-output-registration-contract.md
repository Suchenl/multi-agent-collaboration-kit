# Image Output Registration Contract v3.2.5

Target paper images must be generated raster files with `.png`, `.jpg`, `.jpeg`, or `.webp` extensions, and they must come from the runtime-locked image-generation route in `references/image-generation-route-hardening-policy-v325.md`.

Register:

- S2 sketch images under the S2 candidate registry;
- S5 formal candidate images under the S5 candidate registry;
- S2/S5 explicit rerun images as replacements for the reruned candidate's active image path;
- After S5, human decisions are outside this assistant workflow.
- After S5, human decisions are outside this assistant workflow.

Do not register SVG/HTML/Mermaid/canvas/PPT/PDF/code-drawn/local-rendered outputs as target images. A PNG/JPG/WebP file is invalid when it was produced by Python/PIL/Pillow, Matplotlib, Plotly, Graphviz, TikZ/LaTeX, Mermaid, browser/canvas screenshots, SVG-to-PNG, PPT/PDF rendering, or any other programmatic/local renderer.

Every generated target-paper image registration must cite a matching `image_generation_event_id` whose event records:

- `environment`;
- canonical `generator`: `image_gen`, `create-image`, or `approved-image-api`;
- `required_generator` for that environment;
- `route_guard_status=passed`;
- one or more existing project-run-relative generated raster paths.

After S5, human decisions are outside this assistant workflow.


## Stage-Local Active Image Mirror

After S5, human decisions are outside this assistant workflow.

If the runtime image route returns a raster in an external generator folder, copy the exact generated bytes into the registered stage-local `active_image_path` and record the source generated path, generation event id, checksum if available, and copy/mirror status. This byte-copy mirror is allowed because it preserves the approved generated raster; it is not a local renderer, retouch, screenshot, SVG conversion, or programmatic substitute.

A checkpoint may additionally contain a provenance escrow folder such as `registered-generated-images/`, but escrow is supplemental. Reports, prompt indexes, candidate registries, and user-facing artifact links should prefer the stage-local active path. If a generated image exists only in escrow or only in the generator folder while the registry's active stage-local path is missing, record `missing_stage_local_image_mirror` and do not call the candidate fully registered until the mirror is restored or the registry is explicitly updated.

Required mirror fields for image-generation registration:

```yaml
image_stage_local_mirror:
  source_generated_path: <path or generated file reference>
  active_stage_local_path: <project-run-relative active path>
  byte_copy_only: true
  source_generation_event_id: <event id>
  checksum_match_or_copy_recorded: true | false | unavailable_with_reason
  stage_local_path_exists: true | false
```

Future planned images belong in `pending_future_images`; already-existing generated images are required checkpoint assets. Direct artifact deletion is prohibited for active target-paper image artifacts. Use stale marking, candidate reset, or explicit rewind-step cleanup for reruns.
